# Kubernetes Basics — описание WARC-архива

- **Сайт / раздел:** [Kubernetes Basics](https://kubernetes.io/docs/tutorials/kubernetes-basics/)
- **WARC-файл:** `kubernetes-basics.warc.gz`
- **Дата и время сбора:** ноябрь 2025
- **Размер архива:** ≈ 35.5 mb

## 1. Краткая характеристика раздела

Kubernetes Basics — это официальный вводный туториал по Kubernetes на сайте документации проекта. Раздел состоит из серии модулей, которые по шагам объясняют ключевые концепции оркестрации контейнеров:

- что такое кластер Kubernetes и из чего он состоит;
- как развернуть приложение (Deployment, Pod);
- как организовать доступ к приложению (Service);
- как масштабировать и обновлять приложения;
- как исследовать состояние кластера и отлаживать проблемы.

Материалы оформлены как учебный курс: короткие текстовые объяснения, диаграммы основных сущностей и навигация по модулям. На странице активно используются внутренние ссылки на другие разделы документации (`Concepts`, `Tasks`, `Reference`), а также ссылки на локализованные версии туториала (включая русскую).

Контент в основном текстовый (HTML-страницы), дополненный SVG-диаграммами и стандартными статическими ресурсами (CSS, JS). Раздел — часть статического сайта документации Kubernetes, который собирается и публикуется через Netlify.

## 2. Параметры сбора архива

Для архивации раздела использовался `wpull` с настройками, аналогичными другим сайтам коллекции. Пример команды:

```bash
wpull https://kubernetes.io/docs/tutorials/kubernetes-basics/ \
  --warc-file=kubernetes-basics \
  --recursive --level=1 \
  --page-requisites \
  --no-parent \
  --no-robots
````

Такая конфигурация позволяет:

* **захватить сам раздел Kubernetes Basics** и связанные с ним страницы первого уровня (отдельные модули, режим печати и т. д.);
* **подтянуть все необходимые статические ресурсы**, которые нужны для корректного отображения (CSS, JS, SVG-диаграммы модулей);
* **ограничить глубину обхода**, чтобы не уходить в большую часть общей документации и внешние сайты;
* проигнорировать ограничения `robots.txt` в учебных целях (в реальных проектах цифрового сохранения так делать не рекомендуется).

## 3. Оценка архивируемости (ArchiveReady)

Для URL [https://kubernetes.io/docs/tutorials/kubernetes-basics/](https://kubernetes.io/docs/tutorials/kubernetes-basics/) выполнена проверка в сервисе ArchiveReady (отчёт от 26.11.2025). Итоги: 

* **Website Archivability:** 80 %
* **Accessibility:** 60 %
* **Cohesion (связность):** 90 %
* **Metadata:** 100 %
* **Standards Compliance:** 72 %

### 3.1. HTTP-заголовки и кэширование

Из отчёта ArchiveReady:

* Корректно задан тип содержимого: `Content-Type: text/html; charset=UTF-8`.
* Используется сжатие (`Content-Encoding: gzip`), сервер — Netlify.
* Настроены кэш-заголовки: `Cache-Control: public,max-age=0,must-revalidate`, а также `ETag`, который позволяет ботам не перекачивать неизменённые ресурсы.
* Включён `Strict-Transport-Security`, работает только HTTPS.

Это всё повышает оценку по блокам **Metadata** и **Standards Compliance** и делает сайт предсказуемым с точки зрения поведения в архивации.

### 3.2. robots.txt и карта сайта

В `robots.txt` Kubernetes есть ограничения на часть разделов:

```text
Disallow: /legacy/
Disallow: /v1.0/
Disallow: /v1.1/
Disallow: /404/
Disallow: /404.html
Disallow: /docs/reference/kubernetes-api/
```

Для раздела Kubernetes Basics это не критично, но в целом наличие `Disallow` снижает оценку ArchiveReady по **Accessibility**.

При этом:

* **в robots.txt нет ссылки на sitemap.xml** — за это отчёт даёт отдельное замечание;
* сама карта сайта существует по адресу `https://kubernetes.io/sitemap.xml` и успешно находится ArchiveReady.

Для разового захвата конкретного раздела это не мешает, но делает автоматическую навигацию по всему сайту менее удобной.

### 3.3. Изображения и связность

В отчёте указано, что для анализируемой страницы:

* найдено **6 локальных изображений** и **0 удалённых (remote)**;
* все локальные изображения — это SVG-диаграммы модулей Kubernetes Basics:

  * `module_01.svg` … `module_06.svg` в подкаталоге `public/images/`.

Отсутствие внешних (remote) картинок и использование только локальных SVG-ресурсов даёт высокую оценку по **Cohesion (90 %)**: визуальная часть раздела полностью контролируется самим сайтом, а не внешними хостингами, и хорошо «захватывается» в WARC.

### 3.4. Проверка изображений (JHOVE)

ArchiveReady прогоняет все 6 SVG-файлов через JHOVE:

* формат определяется как **XML 1.0**, кодировка UTF-8;
* корневой элемент — `svg` с пространством имён `http://www.w3.org/2000/svg`;
* статус всех файлов — **Well-Formed** (корректно сформированные XML-документы).

Это означает, что диаграммы модулей пригодны для долгосрочного хранения и последующего рендеринга стандартными SVG-движками.

### 3.5. Ошибки HTML/CSS и производительность

В отчёте также отмечены:

* **ошибки в CSS** для стилей сайта (`style_tutorials.css`, `main.min.…css`) — валидатор W3C фиксирует нарушения спецификаций (конкретные логи см. по ссылкам в отчёте);
* **8 ошибок в HTML** самой страницы (`aria-labelledby` на неподходящих `<div>`, дублирующиеся `id`, некорректные `itemprop` и т. п.);

С точки зрения архивации это не критично: страница успешно загружается, структура HTML понятна парсерам, а CSS-ошибки не мешают извлечению контента.

### 3.6. Ссылочная структура

На странице Kubernetes Basics:

* обнаружено **862 валидные ссылки** и **0 невалидных** (по данным ArchiveReady);
* значительная часть этих ссылок — внутренняя навигация по документации Kubernetes, примерам, справочному разделу и локализованным версиям.

Это делает раздел удобной «точкой входа» для демонстрации того, как веб-архиватор обходит и индексирует большую документацию, начиная с одного туториала.

## 4. Метаданные WARC (metawarc)

Метаданные по объектам архива извлечены командой:

```bash
metawarc metadata --output metadata/kubernetes-basics_meta.jsonl warcs/kubernetes-basics.warc.gz
```

Файл `kubernetes-basics_meta.jsonl` содержит JSON-объекты для каждой WARC-записи и, в частности, следующие поля:

* `url` — адрес ресурса (страницы модулей, режим печати, CSS, JS, SVG-диаграммы и др.);
* `mime` — определённый MIME-тип (`text/html`, `text/css`, `application/javascript`, `image/svg+xml` и т. д.);
* `status` — HTTP-код (200, 301, возможные редиректы);
* служебные поля WARC: `warc_type`, `warc_filename`, `record_length`, идентификаторы записи и пр.

По типам содержимого в архиве преобладают:

* **`text/html`** — основная HTML-страница раздела и связанные с ней страницы (модули, печатная версия);
* **`text/css`** — таблицы стилей `style_tutorials.css`, скомпилированные SCSS-стили;
* **`application/javascript`** — фронтенд-скрипты навигации, меню, аналитики;
* **`image/svg+xml`** — диаграммы модулей Kubernetes Basics.

Такая структура хорошо подходит для учебного анализа:

* можно строить таблицы распределения по MIME-типам и расширениям;
* легко выделить только HTML-страницы (для последующего текстового анализа);
* можно отдельно рассмотреть, как в архив попали SVG-диаграммы и какие параметры у них сохранились.

При необходимости на основе `kubernetes-basics_meta.jsonl` можно сформировать агрегированные таблицы и включить их в общий `warc-metadata-summary.md` (аналогично тому, как это сделано для других сайтов коллекции).

## 5. Наблюдения и выводы

* Раздел **Kubernetes Basics** представляет собой хорошо структурированный, связный и относительно самодостаточный учебный ресурс, что делает его удобным кандидатом для включения в коллекцию WARC.
* **Высокая связность (Cohesion 90 %) и отсутствие внешних изображений** означают, что почти всё визуальное оформление и навигация полностью контролируются сайтом, а значит, хорошо попадают в архив.
* **Метаданные и HTTP-заголовки настроены корректно (Metadata 100 %, Standards 72 %)**, что облегчает работу архивирующих ботов и позволяет оптимизировать повторные заходы (через `ETag` и кэш-заголовки).
* Наличие **роботовских запретов в robots.txt** и отсутствие ссылки на sitemap.xml ухудшают формальный показатель Accessibility (60 %), но в рамках учебной задачи с использованием `--no-robots` это не мешает получить полный срез раздела.
* Ошибки в HTML и CSS фиксируются валидаторами, но **не мешают ни отображению, ни архивации**: страница успешно загружается, а структурированная документация и ссылки остаются доступными.
* В целом раздел демонстрирует **хорошую архивируемость (80 %)** и служит примером того, как сложный документационный сайт с богатой навигацией и структурой может быть захвачен и проанализирован с помощью связки `wpull` + `metawarc` + ArchiveReady.
