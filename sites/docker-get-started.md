## 1. Краткая характеристика раздела

Раздел **Get started** — это входная точка документации Docker для новичков. Он объясняет, что такое Docker и контейнеры, как установить окружение и как постепенно перейти от простой команды `docker run` к работе с образами, Dockerfile и multi-container-приложениями.

По структуре (что видно и по списку ссылок в отчёте ArchiveReady) раздел делится на несколько логических блоков: :contentReference[oaicite:0]{index=0}  

- **Introduction** — общие принципы, зачем нужны контейнеры, чем Docker отличается от виртуальных машин, какие есть базовые сущности: image, container, registry.
- **Foundations of Docker / Docker concepts** — серии коротких статей «what is a container / image / registry / docker compose», «building images», «multi-stage builds», «publishing ports», «persisting data» и т.п.
- **Workshop** — пошаговый практический курс (шаги 02–10), где пользователь:
  - собирает и запускает тестовое приложение;
  - обновляет образ и выкатывает изменения;
  - делится приложением через registry;
  - добавляет тома (bind mounts) и работу с данными;
  - переходит к multi-container и Docker Compose;
  - знакомится с best practices для образов.
- **Resources / manuals / guides** — ссылки на дальнейшее чтение, референс и руководства.

Отдельные ссылки ведут на страницы установки Docker Desktop, pricing-страницу и официальный сайт компании, а также на социальные сети Docker (YouTube, X, Facebook, LinkedIn, Instagram). Таким образом, раздел «Get started» выступает как хаб, соединяющий базовый туториал, концепции и навигацию по всей документации.

## 2. Параметры сбора архива

Для архивации использовалась утилита `wpull` с настройками, согласованными с другими сайтами коллекции. Пример команды:

```bash
wpull https://docs.docker.com/get-started/ \
  --warc-file=docker-get-started \
  --recursive --level=1 \
  --page-requisites \
  --no-parent \
  --no-robots
````

Такой режим:

* захватывает саму страницу **Get started** и страницы первого уровня, на которые она ссылается (introduction, workshop-шаги, базовые понятия, overview, install-страницы);
* подтягивает необходимые статические ресурсы (`style.…css`, `scripts.js`, шрифты и т.п.) для корректного отображения;
* ограничивает глубину обхода, чтобы не тянуть всю документацию Docker целиком;
* игнорирует ограничения `robots.txt` (это допустимо в рамках учебного задания, но не рекомендуется для реального веб-архивирования).

### 3.1. HTTP-заголовки и кэширование

В отчёте зафиксированы следующие ключевые HTTP-заголовки: 

* `Content-Type: text/html` — тип содержимого определён корректно.
* `Content-Encoding: gzip` — используется сжатие.
* `Server: AmazonS3` + `X-Cache: Hit from cloudfront`, `Via: …cloudfront.net` — раздача идёт через связку S3 + CloudFront.
* `Last-Modified: Fri, 05 Dec 2025 13:58:26 GMT` и `ETag: W/"ca433…"` — присутствуют заголовки для условных запросов.
* `Age: 2862` — ответ взят из кэша CDN.
* `Strict-Transport-Security: max-age=31536000`, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff` — современные заголовки безопасности.

ArchiveReady подчёркивает, что **HTTP caching headers включены**: наличие `Last-Modified` и `ETag` помогает архивирующим ботам избегать повторной загрузки неизменённого контента, что положительно сказывается на метриках **Metadata** и **Accessibility**.

### 3.2. robots.txt и карта сайта

В `robots.txt` Docker Docs присутствуют: 

* команда `Disallow: /unassociated-machines/`, которая частично снижает оценку **Accessibility** (формально есть закрытая зона);
* при этом в `robots.txt` **явно указана ссылка на `sitemap.xml`**, что положительно оценивается сервисом;
* файл карты сайта доступен по адресу `https://docs.docker.com/sitemap.xml` и учитывается в расчёте рейтинга.

Таким образом, для раздела **Get started** формально нет прямых ограничений, а наличие sitemap облегчает обход всего сайта.

### 3.3. Изображения и связность

Отдельно отмечено, что **на самой странице `get-started/` изображения не обнаружены**: 

> No images found in this page. Therefore, it is easier to be archived.

Это даёт 100 % по соответствующей подметрике Cohesion: отсутствие встроенных картинок (особенно внешних) упрощает захват содержимого и делает страницу менее зависимой от сторонних ресурсов. Визуальное оформление реализовано через CSS, иконки и шрифты.

### 3.4. CSS и скрипты

С точки зрения статики отчёт показывает: 

* **CSS:** 1 локальный файл стилей
  `https://docs.docker.com/css/style.…css` (remote CSS — 0).
  Это хорошо влияет на **Cohesion**: оформление контролируется самим сайтом, а не внешними CDN.
* **JS:** 1 локальный скрипт `https://docs.docker.com/scripts.js` и **2 внешних скрипта**:

  * `https://cdn.cookielaw.org/scripttemplates/otSDKStub.js` (cookie/consent баннер);
  * `https://widget.kapa.ai/kapa-widget.bundle.js` (виджет помощи/чата).

Наличие внешних JS снижает оценку по **Cohesion**: при архивации такие ресурсы могут не попасть в WARC, а в будущем — перестать существовать, что повлияет на поведение интерактивных элементов (cookie-баннер, чат-виджет).

Отдельно ArchiveReady ругается на **8 inline script elements**: это считается плохой практикой и усложняет архивацию, так как логика частично «зашита» непосредственно в HTML, а не вынесена в отдельные файлы.

Формально это снижает **Standards Compliance** до 60 %, хотя в реальных браузерах страница отображается нормально.

Также есть противоречивые данные по времени ответа:

Для учебной задачи это можно трактовать как пример того, что измерения ArchiveReady зависят от момента и условий проверки, но в любом случае сайт доступен и полностью отдается.

## 4. Метаданные WARC (metawarc)

На момент подготовки этого файла **`docker-get-started_meta.jsonl` пуст**, то есть анализ содержимого WARC-архива с помощью `metawarc metadata` ещё не выполнялся (или его результат не был сохранён).

Ожидаемая команда для извлечения метаданных:

```bash
metawarc metadata \
  --output metadata/docker-get-started_meta.jsonl \
  warcs/docker-get-started.warc.gz
```

После её выполнения в `docker-get-started_meta.jsonl` появятся записи по каждому объекту WARC (HTML-страницы, CSS, JS, возможные шрифты и т.д.) со стандартными полями:

* `url` — адрес ресурса;
* `mime` — тип содержимого (ожидаемо будут преобладать `text/html`, `text/css`, `application/javascript`);
* `status` — HTTP-код (200, 301/302 для редиректов);
* служебные поля (`warc_type`, `warc_filename`, `record_length`, идентификаторы записей).

На основе этого файла можно будет:

* построить таблицу распределения по MIME-типам и расширениям (как для других сайтов коллекции);
* оценить долю HTML-страниц vs статических ресурсов;
* при желании — включить агрегированную статистику в общий `warc-metadata-summary.md`.

Пока же метаданные по Docker Docs в итоговую сводку не попадают — это отдельный шаг, который нужно выполнить позже.

## 5. Наблюдения и выводы

* Раздел **Get started** в Docker Docs — это хорошо структурированная обучающая точка входа, объединяющая концепции, практический воркшоп и ссылки на дальнейшие материалы. С точки зрения веб-архивирования он даёт компактный, но содержательный срез документации.
* **Высокая оценка Metadata (100 %) и корректные HTTP-заголовки** (тип содержимого, сжатие, Last-Modified, ETag, HSTS, X-Frame-Options, X-Content-Type-Options) упрощают работу архивирующих ботов.
* **Когезия (78 %) и отсутствие изображений на самой странице** означают, что основной контент — текст + CSS, а не медиафайлы. При этом зависимость от внешних JS-скриптов (cookie-баннер, виджет поддержки) может привести к частичной потере интерактивности в офлайн-копии.
* **Наличие sitemap и ссылки на него в robots.txt** улучшает формальную доступность сайта для обхода, несмотря на частичный `Disallow` для отдельных разделов.
* Ошибки HTML/CSS и использование inline-скриптов снижают формальные показатели стандартов, но практически не мешают ни отображению, ни архивации содержимого.
* Чтобы довести документацию Docker в коллекции до уровня других сайтов, необходимо **дозапустить `metawarc metadata` для `docker-get-started.warc.gz`** и включить полученную статистику в общую сводную таблицу WARC-архивов.
